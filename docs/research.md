# Исследование: технические решения для проверки сайтов на соответствие 152-ФЗ

## 1. Кто уже решил эту проблему

### Российские сервисы (прямые конкуренты)

| Сервис | Модель работы |
|---|---|
| [СайтСкан.ру](https://saitscan.ru/) | Онлайн-проверка по URL, ориентир на 152-ФЗ, отчёт с нарушениями. Серверы в РФ. |
| [ФЗ-152 Аудит (rffz.ru)](https://rffz.ru/) | Заявленное время аудита — 2 минуты по одному URL. |
| [1ps.ru калькулятор](https://1ps.ru/instrumentyi-i-kalkulyatoryi/proverka-sajta-na-sootvetstvie-152-fz/) | Часть SEO-сервиса, проверка по чек-листу. |
| [pobedaprava.ru](https://pobedaprava.ru/pd/audit-152fz/) | Юридический аудит — больше ручной, чем автомат. |

Все они закрыты по стеку (SaaS), но по поведению видно общий подход: краулинг сайта → проверка по фиксированному чек-листу (cookie, политика конфиденциальности, форма согласия, контактные данные, формы со сбором ПДн, счётчики и пиксели) → отчёт.

Косвенно полезный официальный источник — реестр РКН [eais.rkn.gov.ru](https://eais.rkn.gov.ru/en/) и [сервис проверки блокировок](https://blocklist.rkn.gov.ru/) — это только статус блокировки, а не аудит. Своего автоматического сканера соответствия РКН не предоставляет.

### GDPR-аналоги (open source — ценны как референс архитектуры)

| Проект | Что даёт |
|---|---|
| [dev4privacy/gdpr-analyzer](https://github.com/dev4privacy/gdpr-analyzer) | Анализирует исходник страницы + поведение в браузере + безопасность канала. Готовая модель «что проверять». |
| [CovenantSQL/CookieScanner](https://github.com/CovenantSQL/CookieScanner) | Сканер cookies с генерацией отчёта. |
| Termly / Cookiebot / 2GDPR | Коммерческие, но публикуют методологии. |

**Что взять от GDPR-инструментов:** разделение проверок на три слоя — статический HTML/HTTP, динамическое поведение в браузере, сетевые запросы (cookies, трекеры). Этот же скелет работает для 152-ФЗ, меняется только корпус правил.

## 2. Подход, который считается best practice сегодня

Сходимость по индустрии (2026) выглядит так:

1. **Headless-браузер вместо чистого HTTP-скрапа.** Современные сайты грузят согласия и формы JS-ом — без рендеринга проверка пропустит половину нарушений. Консенсус — [Playwright](https://www.browserstack.com/guide/playwright-vs-puppeteer) (мультибраузер, auto-wait, Python+JS API, активная разработка). Puppeteer — узкий Node/Chromium вариант, Selenium считается устаревшим для нового кода.
2. **Корпус правил — структурированные файлы, движок-интерпретатор.** Это уже сделано в [docs/laws/schema.md](docs/laws/schema.md): Markdown + YAML-фронтматтер, блок `detection` с `page_signals` / `site_signals` (CSS-селекторы, ключевые слова, имена check-функций), `penalties`, `recommendation`. Парсится [python-frontmatter](https://github.com/eyeseast/python-frontmatter) + валидируется Pydantic-моделями ([стандартный паттерн 2026](https://medium.com/better-programming/validating-yaml-configs-made-easy-with-pydantic-594522612db5)).
3. **Сначала детерминированные правила, LLM — точечно.** Большинство нарушений (политика отсутствует, форма без чекбокса, cookie без согласия, нет данных оператора, HTTP вместо HTTPS) описываются конечным набором сигналов и ловятся без LLM. Семантические проверки (раскрыты ли цели обработки в политике, корректна ли формулировка согласия) — единственное, где LLM реально нужен, и их можно вводить итеративно.
4. **PDF из HTML — два лагеря.** [WeasyPrint](https://www.nutrient.io/blog/how-to-generate-pdf-reports-from-html-in-python/) для статических отчётов (Python-нативный, без Chromium, маленький вывод) и Puppeteer/Playwright PDF — когда нужен JS-рендер графиков. Для отчёта об аудите динамика не нужна.
5. **Очереди — не для MVP.** Встроенные `BackgroundTasks` FastAPI ненадёжны при рестарте, [ARQ + Redis](https://davidmuraya.com/blog/fastapi-background-tasks-arq-vs-built-in/) и Celery — следующий шаг. Но в MVP результат не сохраняется и пользователь ждёт в той же сессии — нужна одна async-таска внутри одного процесса + SSE для прогресса. Никаких брокеров.

## 3. Сравнение ключевых альтернатив

### Браузерный движок

| Решение | Плюсы | Минусы |
|---|---|---|
| **Playwright (Python)** | Мультибраузер, auto-wait, тот же язык, что и бэкенд, активно развивается | Чуть тяжелее Puppeteer |
| Puppeteer (Node) | Быстрый старт, маленькая зависимость | Только Chromium, только Node — раздваивает стек |
| Selenium | Универсален | Медленный, шумный API, считается устаревшим для новых проектов |

### LLM-провайдер (если/когда понадобится)

Уточнение по итогам доп. проверки: бесплатный тариф [GigaChat](https://developers.sber.ru/docs/ru/gigachat/api/tariffs) — **1 млн токенов в год** (не в месяц) и single-thread. Для публичного сервиса, где LLM зовётся на каждой проверке, лимита хватит на пару тысяч сканирований при разумных промптах. Этот лимит и определяет реальный выбор.

| Решение | Бесплатность | Русский юр-контекст |
|---|---|---|
| **GigaChat (Sber)** | 1M токенов / год для физлиц, single-thread | Обучен на нормативке РФ, в т.ч. законах и подзаконных актах |
| YandexGPT | Бесплатные кредиты Yandex Cloud, потом платно | Сильный русский, чуть слабее по правовой узкоспециализации |
| Claude / GPT-4o | Платно | Лучшее качество, но требует прокси из РФ — иронично для проверки РФ-законов |
| Локально (Ollama + Qwen 2.5 / Saiga) | Полностью бесплатно | Нет потолка по токенам; нужен сервер с 16+ ГБ RAM / GPU, латентность выше |

**Вывод:** в MVP LLM не критичен — корпус справочника спроектирован так, что детектирование делается сигналами, а тексты `description` / `recommendation` / `penalties` уже написаны людьми и идут в отчёт напрямую. LLM подключаем позже, по конкретным семантическим проверкам, когда они появятся.

### Генерация PDF

| Решение | Плюсы | Минусы |
|---|---|---|
| **WeasyPrint** | Чистый Python, поддержка CSS Paged Media, вывод 8 КБ vs 16 КБ у Playwright, без Chromium | Не выполняет JS — для нашего отчёта это не нужно |
| Playwright/Puppeteer PDF | Полноценный рендер, графики из Chart.js | Тяжёлая зависимость (Chromium), уже стоит для краулинга — можно переиспользовать |
| ReportLab | Нативный PDF API | Писать вёрстку императивно — медленно и больно |

### Архитектура очереди для MVP

| Решение | Когда уместно |
|---|---|
| **FastAPI async + SSE/WebSocket для прогресса** | MVP, результат не сохраняется, одна задача = один запрос пользователя |
| ARQ + Redis | Когда задач много, нужна устойчивость к рестартам, отложенная отдача |
| Celery + Redis/RabbitMQ | Сложные пайплайны, ретраи, распределённость — оверкилл для MVP |

## 3a. Влияние ADR-0001 на стек

Прочтение [docs/adr/0001-laws-reference-structure.md](adr/0001-laws-reference-structure.md) добавляет к стеку несколько артефактов, но не меняет принципиальный выбор:

1. **`index.yml` — генерируется из фронтматтеров.** Нужен CLI-скрипт `tools/rebuild_index.py`: проходит по `docs/laws/**/*.md`, собирает идентификацию + теги в один индекс. Это часть сборки корпуса, а не runtime.
2. **Линтер корпуса.** Проверки целостности из ADR-0001 (валидность YAML, обязательные поля, уникальность `id` нарушений, существование `related`-ссылок и `[[id]]` в теле, `verified_at` старше 6 мес → warning) выполняются тем же `tools/rebuild_index.py` — отдельный линтер не нужен.
3. **Поддержка подкаталогов** (`docs/laws/common/`) при загрузке — `glob("**/*.md")` вместо `glob("*.md")`. Тривиально.
4. **152-ФЗ как эталон — 754 строки.** Это меняет прикидку нагрузки: при ~17 актах корпус — порядка 5–10 тыс. строк YAML+MD. Парсинг и валидация на старте остаются миллисекундными, в памяти десятки килобайт. Никакого SQLite/индексирования не нужно.
5. **TODO из ADR не блокируют стек.** Открытые вопросы по нумерации частей ст. 13.11 КоАП и заглушкам ID — задачи для агентов-сборщиков корпуса, движок проверки на них не завязан.

Вывод: ADR-0001 закрепляет именно ту схему, под которую и собирался стек. Один процесс, парсер + линтер + индекс-генератор как утилиты, никаких новых компонентов.

## 3b. Что реально ловится без LLM, а что нет

Честный разбор по типам нарушений из MVP-корпуса:

### Полностью детерминированно (≈ 65% нарушений)

| Тип проверки | Как ловится |
|---|---|
| Наличие/отсутствие страницы политики ПДн | `lookup_pages_by_keywords` + анализ ссылок навигации/футера |
| Форма содержит поля ПДн (email/телефон/имя) | CSS-селекторы из `html_patterns` |
| Чекбокс согласия рядом с формой | DOM-обход + ключевые слова в `<label>` (`согласие`, `обработку`, `персональных данных`) |
| Cookie выставлены до согласия | CDP-логирование cookies + момент клика по баннеру (Playwright) |
| HTTPS / HSTS / mixed content | Заголовки ответа |
| Реквизиты продавца (наименование, ОГРН, адрес) | regex + поиск в футере/контактах |
| Маркер возрастной категории (0+, 6+, 12+, 16+, 18+) | regex по DOM |
| `lang="ru"` / `<html lang>` | Атрибут DOM |
| Локализация ПДн (242-ФЗ) — куда улетают формы/аналитика | Анализ origin'ов сетевых запросов + список known-foreign доменов |
| Erid-токен в рекламных блоках (когда блок размечен) | regex по data-атрибутам / URL |
| Cookie-banner — наличие, наличие кнопки «Отклонить» | DOM-обход |
| Уведомление РКН о работе с ПДн | **Не детектируется снаружи** — это информационный блок в отчёте, без сигналов |

### Требует семантики (≈ 35% — без LLM либо честно отказываемся, либо даём ложные срабатывания)

| Тип проверки | Почему сигналами не закрыть |
|---|---|
| Текст согласия раскрывает цели, состав ПДн, срок хранения, операторов | Бесконечно много формулировок; ключевые слова дают много ложных срабатываний |
| Политика конфиденциальности содержит все обязательные разделы по ст. 18.1 152-ФЗ | То же самое — структура у всех своя |
| Идентификация *что является рекламой* для маркировки по 38-ФЗ | Семантическая граница реклама/контент — даже для людей спорная |
| Корректность формулировок cookie-banner (не dark pattern) | Визуальная иерархия + смысл текста |
| Иностранные слова без перевода (53-ФЗ) | Нужен лексический анализ + контекст |
| Соответствие контента возрастной категории | По сути требует прочтения и понимания контента |
| Соответствие условий продажи / оферты / возврата требованиям ЗПП | Юридический разбор текста |

### Покрытие в зависимости от LLM

| Вариант | Покрытие | Стоимость |
|---|---|---|
| Pure-deterministic | ~65% | бесплатно |
| **Гибрид с платным GigaChat** | **~90–95%** | **~150–300 ₽ за 1M токенов** |
| Гибрид с локальным Saiga / Qwen 2.5 через Ollama | ~85–90% | бесплатно, но нужен сервер с ≥ 16 ГБ RAM |
| Гибрид с Claude Sonnet | ~95%+ | ~900 ₽ за 1M токенов + прокси из РФ |

Для проекта «для себя, с небольшим бюджетом, мало пользователей» 65% неприемлемо — экономия на LLM не оправдана. Базовая прикидка: один скан = политика 30–80 КБ + cookie-banner + текст оферты + до 10 семантических промптов ≈ **20–50 тыс. токенов на скан**. При 10 сканах в неделю это ~2–5M токенов в месяц, по тарифам GigaChat — **300–1500 ₽/мес**. Это и есть «небольшая сумма».



**Выбранный стек (MVP, гибрид с платным LLM):**

| Слой | Решение |
|---|---|
| Бэкенд | **Python 3.12 + FastAPI** (один процесс, без брокеров) |
| Браузер | **Playwright (Python)** в headless-режиме |
| Парсинг корпуса | **python-frontmatter** + **Pydantic** v2 (модели зеркалят [docs/laws/schema.md](docs/laws/schema.md)) |
| Утилиты корпуса | `tools/rebuild_index.py` — пересборка `index.yml` и проверка целостности кросс-ссылок (объединённая команда) |
| Движок проверки | Гибрид: исполнитель детерминированных сигналов (`page_signals`, `site_signals`) + семантические проверки через LLM-адаптер |
| LLM | **GigaChat (платный)** как primary, с интерфейсом-адаптером, чтобы поменять на YandexGPT / локальный Saiga при необходимости |
| LLM-кэш | Локальный SQLite-кэш по `sha256(промпт + текст)` — повторные проверки одного сайта/политики не тратят токены |
| PDF | **WeasyPrint** + Jinja2-шаблон отчёта |
| Прогресс в UI | **SSE (Server-Sent Events)** из FastAPI |
| Фронтенд | **Next.js (App Router) + Tailwind** |
| Хостинг | Фронт — **Vercel free tier**; бэк — **Fly.io / Render free tier** (Playwright требует свой Chromium) |

### Почему именно так

**Почему GigaChat, а не Claude / GPT-4o.** Сравнительно: GigaChat обучен на русской нормативке (включая законы и подзаконные акты), API доступен из РФ без прокси, цена в 3–6 раз ниже западных моделей. Claude качественно сильнее на длинных юридических текстах, но для проекта *про российское законодательство* зависимость от прокси выглядит хрупко и дороже без пропорционального выигрыша. YandexGPT — близкая альтернатива, по обзорам чуть слабее именно в правовой узкоспециализации; держим как фоллбек через тот же адаптер.

**Почему не локальный Ollama.** Бесплатно по токенам, но контейнер с моделью на 8–16 ГБ RAM не входит во free-tier хостинга (Fly.io/Render) — нужна платная VPS от ~500–1000 ₽/мес или своя домашняя машина с пробросом. Это сопоставимо или дороже, чем 300–1500 ₽/мес за GigaChat при значительно худшей удобности.

**Почему не Node.js + Puppeteer.** Python выигрывает по трём осям: (1) `python-frontmatter` + Pydantic дают валидацию корпуса из коробки; (2) WeasyPrint легче PDF-через-Chromium; (3) GigaChat / YandexGPT / Saiga — все имеют первоклассные Python SDK. Раздваивать стек смысла нет.

**Почему один процесс FastAPI, а не Celery / ARQ + Redis.** Скан занимает 20–60 сек, в MVP результат не сохраняется, пользователь ждёт в той же сессии. Async-функция + SSE даёт нужную «анимацию проверки» без брокера. Когда добавится история проверок — переезд на ARQ занимает день.

**Сравнение с альтернативой «Celery / ARQ + Redis с первого дня».** В MVP результаты не сохраняются, пользователь ждёт в той же сессии — выделенный воркер-пул не оправдан. Одна async-функция FastAPI + SSE даёт нужную «анимацию проверки» без отдельной инфраструктуры. Переезд на ARQ потом — один день, если бизнес-логика отделена от транспорта.

### Что учесть при имплементации

1. **Pydantic-модель корпуса = единственный источник правды по схеме.** Один файл `models.py` описывает структуру закона и нарушения 1-в-1 с [docs/laws/schema.md](docs/laws/schema.md). При парсинге каждый файл валидируется; падение валидации = ошибка сборки, не runtime. Это же даст автогенерацию JSON Schema для редактора корпуса позже.
2. **Реестр check-функций.** Поле `detection.site_signals[].check` (например, `lookup_pages_by_keywords`) — это имя функции в коде. Заводим словарь `{name → callable}`, добавление новой проверки = новая функция + одна строка в реестре. Никаких eval / динамической загрузки.
3. **Корпус — артефакт сборки.** Парсить `docs/laws/*.md` один раз на старте и держать в памяти; не читать с диска на каждый запрос. При изменении файла — перезапуск процесса (для MVP достаточно).
4. **Playwright + хостинг.** Vercel/Netlify под бэк не подойдут. Контейнерный хост (Fly.io / Render / своя VPS). В Dockerfile — базовый образ `mcr.microsoft.com/playwright/python:v1.49.0-noble` (или актуальный), экономит часы установки Chromium и системных шрифтов.
5. **robots.txt и нагрузка на чужие сайты.** Уважать robots.txt, таймаут 15–30 сек, не больше N страниц на домен (10–20 для MVP). Один User-Agent с явной идентификацией сервиса — иначе нарвёмся на жалобы и блокировки.
6. **SSE-канал для анимации «что проверяем сейчас».** Каждый шаг (загружаем главную → ищем cookie-banner → читаем политику → проверяем формы → ...) пушит событие. Имена шагов — те же, что в UI; источник этих имён — категории сигналов из корпуса (`form_without_consent_checkbox` → «Проверяем формы сбора ПДн»).
7. **PDF — без сторонних шрифтов через CDN.** WeasyPrint грузит ресурсы при рендере; шрифты класть локально, иначе отчёт собирается дольше и хрупче. Шаблон Jinja2 итерирует по найденным `violations` с уже готовыми `description`, `penalties`, `recommendation` — нулевая логика в шаблоне.
8. **Дисклеймер в каждом отчёте.** «Автоматическая проверка не заменяет юридическое заключение». Снимает риск претензий и оставляет место под платную услугу «проверка юристом» позже.
9. **LLM-адаптер с одним интерфейсом.** Один класс `LLMProvider` с методом `ask(prompt, context) -> StructuredAnswer`. Реализации: `GigaChatProvider` (primary), `YandexGPTProvider` (фоллбек), `OllamaProvider` (для офлайна). Переключение — через env-переменную, движок про конкретного провайдера ничего не знает.
10. **Семантические сигналы — новый тип `check` в корпусе.** Расширить схему сигналов значениями `check`-функций типа `policy_discloses_purposes`, `consent_text_valid`, `cookie_banner_no_dark_pattern`. Реестр check-функций (см. п. 2) маппит их на код, который собирает текст → промпт → LLM → нормализованный ответ (`pass`/`fail` + цитата + объяснение). Схема в [docs/laws/schema.md](docs/laws/schema.md) расширения не требует, добавляются только новые имена в существующее поле `check`.
11. **LLM-промпты — отдельная папка `prompts/*.md` под версионированием.** Каждый промпт — отдельный файл с frontmatter (имя check-функции, ожидаемая структура ответа, модель). Промпты править отдельно от кода, легко A/B-тестировать.
12. **Структурированный ответ от LLM.** Запрашивать всегда JSON со строгой схемой (`{"verdict": "pass|fail|unclear", "evidence": "...", "explanation": "..."}`). Валидировать Pydantic-ом; при `unclear` или невалидном JSON — помечать нарушение как «требует ручной проверки», не считать pass.
13. **LLM-кэш по хэшу — обязательно.** SQLite-таблица `(prompt_id, content_hash, model, version) → answer`. Политика конфиденциальности у большинства сайтов меняется редко — повторный скан того же URL не должен тратить токены заново. Это же снимает 90% стоимости при отладке. **Инвалидация автоматическая:** изменился текст политики → другой `content_hash` → cache miss → новый запрос. **Перед хэшированием нормализовать:** strip пробелов, схлопнуть пустые строки, вырезать заведомо динамические фрагменты (даты «Последнее обновление: …», копирайт-год, CSRF/nonce). Без этого каждый второй скан даёт ложный cache miss на косметике.
14. **Контроль бюджета.** Простой счётчик использованных токенов в той же SQLite + лимит на скан (например, 50 тыс. токенов максимум). Если упёрлись — оставшиеся семантические проверки помечаем «не выполнено из-за лимита», а не делаем тихий пропуск.
15. **Гонять промпты только на релевантных фрагментах.** Не отдавать LLM всю страницу — сначала вырезать политику/оферту/banner-текст детерминированно, потом подавать в LLM только нужный кусок. Это снижает токены в 5–20 раз.
16. **Целостность корпуса — в CI.** Линтер из ADR-0001 (валидация Pydantic, проверка `related`, `[[id]]`, уникальность violation-id, возраст `verified_at`) запускать на каждый PR.
