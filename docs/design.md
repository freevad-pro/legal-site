# Дизайн UI — Legal Site

Документ фиксирует визуальный язык продукта: референс, тональность, стек, палитру, типографику, структуру single-page, поведение интеракций и a11y-минимум. Опирается на [idea.md](idea.md), [vision.md](vision.md). Является основанием для итерации 6 (Frontend MVP).

**Живой прототип:** [docs/design/preview.html](design/preview.html) — открывается в любом браузере. Демонстрирует hero, форму, прогресс с анимацией, страницу результата и финальный CTA.

## 1. Контекст и аудитория

Продукт — автоматизированная проверка сайтов на соответствие законодательству РФ. ЦА широкая: владельцы сайтов, маркетологи и продакты, веб-студии, юристы, комплаенс-специалисты ([idea.md](idea.md)). Один пользователь — один скан, никаких регистраций и сохранённых отчётов (MVP).

## 2. Референс-вектор

Российский корпоративный B2B-стиль: чистая белая база, зелёный CTA, плоские серые карточки с большим скруглением, плотная подача без декоративных украшений.

Ключевые параметры визуального языка:

- **Брендовый CTA-цвет** — `#26AD50` (насыщенный зелёный «доверия» — частая ассоциация в российских корпоративных сервисах: проверки, согласования, отметки «всё ок»).
- **Композиция**: hero — текст-обещание сверху + крупная зелёная карточка-CTA с формой поиска ниже + серые карточки преимуществ под формой («без теней», `border-radius` 16 px).
- **Шрифт основной**: **Onest** (Google Fonts, бесплатный, кириллица из коробки, современный гротеск). Шрифт моноширинный: **JetBrains Mono** для URL и evidence.

Прежние эксперименты с editorial/serif-стилем (Spectral italic, кремовый фон, латунные акценты, grain-overlay) отклонены — не подходят к B2B-комплаенс контексту.

## 3. Тональность и язык

Сбалансированно-деловая, без официоза и без панибратства.

- Заголовки и пояснения — обычным русским языком («Проверим сайт», «Найдём нарушения»).
- Технические детали (статьи закона, селекторы, цитаты) — в отдельных блоках карточки, не в заголовке.
- Severity именуются по-русски: «Критический», «Высокий», «Средний», «Низкий», «Не проверено». Никаких `critical/high/medium/low` в UI.
- Термин «LLM-провайдер» не используется — пишем «автоматический анализ текста».
- Юридический блок именуется **«Ограничение ответственности»** (общепринятый в российской договорной практике термин), не «дисклеймер» и не «оговорка».
- Эмодзи в UI запрещены. Иконки — только Lucide.

## 4. UI-стек

- **Next.js 15** (App Router) + **TypeScript**, `output: 'export'` (см. [vision.md](vision.md) → Frontend).
- **Tailwind CSS** + **shadcn/ui** (компоненты копируются в проект, без рантайм-зависимости; стили правятся под наши токены).
- **Lucide React** для всех иконок (входит в экосистему shadcn).
- Анимации — `tailwindcss-animate` (включён в shadcn); сторонние motion-библиотеки **не подключаем** в MVP.
- SSE — нативный `EventSource`.

## 5. Палитра

Все цвета вводятся как CSS-переменные / токены в Tailwind config. Не разбрасывать raw-hex по компонентам.

|-------|-----|------------|
| `bg.base` | `#FFFFFF` | Фон страницы и поверхностей первого уровня |
| `bg.soft` | `#F4F4F4` | Карточки преимуществ, серые блоки, плашки |
| `bg.gray` | `#EEEEEE` | Hover-состояние серых карточек |
| `line` | `#ECECEC` | Тонкие границы, разделители |
| `line.strong` | `#D0D0D0` | Hover-границы, активные элементы |
| `ink.primary` | `#1F1F1F` | Основной текст, заголовки |
| `ink.secondary` | `#666666` | Пояснения, мета |
| `ink.muted` | `#ADADAD` | Подсказки, плейсхолдеры, неактивные элементы |
| `ink.faint` | `#D0D0D0` | Декоративные точки, разделители-точки |
| `brand` | `#26AD50` | CTA, акценты, иконки на брендовом фоне |
| `brand.hover` | `#1F9544` | Hover/active для CTA |
| `brand.dark` | `#144942` | Тёмно-зелёный (опционально, для футера если будет тёмный) |
| `brand.soft` | `#E5F4EA` | Светлый фон под зелёный бейдж |
| `link` | `#1F6FBF` | Информационные ссылки, иконка info в плашке уведомления |

### 5.2 Severity-палитра

Каждый уровень = текст + soft-фон + border. Различимы по hue (red→orange→yellow→blue), что снимает риск дальтонизма. Дополнительно дублируем текстовой меткой («Критический», «Высокий» и т. п.) — color не единственный носитель смысла.

| Уровень | Текст | Soft-фон | Border |
|---------|-------|----------|--------|
| Критический | `#C73C3C` | `#FBE9E9` | `#F2BCBC` |
| Высокий | `#D87A1A` | `#FBEDD9` | `#EFCB94` |
| Средний | `#C99A1F` | `#FAF1D2` | `#EDD584` |
| Низкий | `#3E7FBF` | `#E6F0FA` | `#B5D0EB` |
| Не проверено | `#666666` | `#F4F4F4` | `#D0D0D0` |

Левый стрип карточки нарушения (`box-shadow: inset 3px 0 0 var(--stripe)`) — тем же текстовым цветом severity.

### 5.3 Favicon-палитра (цветной кружок с буквой)

Цвет берётся по хешу хоста из массива: `#2563EB / #7C3AED / #DB2777 / #DC2626 / #EA580C / #CA8A04 / #16A34A / #0891B2 / #4F46E5`. Это даёт «продуктовое» ощущение и узнаваемость сайта в шапке отчёта.

## 6. Типографика

- **Onest** (sans, Google Fonts, кириллица + латиница) — основной UI и body. Веса 400 / 500 / 600 / 700 / 800. Подключение через `next/font/google` с `subsets: ['latin', 'cyrillic']`.
- **JetBrains Mono** (Google Fonts, кириллица) — URL, технические цитаты, evidence, номера, mono-числа.

Шкала размеров:

| Применение | Размер | Вес |
|------------|--------|-----|
| h1 (hero) | 34 / 44 / 52 px (mobile / sm / lg) | 700 |
| h2 секции | 24 / 28 px | 700 |
| h3 (карточки, блоки) | 17–20 px | 700 |
| Основной текст | 14–16 px | 400 |
| Eyebrow / метки | 11 px / uppercase / tracking 0.14em | 600 |
| Числа-счётчики | 32–40 px | 700, mono, tabular-nums |
| Подписи / footer | 12–13 px | 400 |

Минимум 16 px для body на mobile, чтобы iOS не делал автозум при фокусе на инпут.

## 7. Скругления и spacing

| Токен | Значение | Применение |
|-------|----------|------------|
| `--radius-card` | 16 px | Крупные карточки, плашки, секции CTA |
| `--radius-btn` | 8 px | Кнопки, инпуты, чипы |
| `--radius-pill` | 999 px | Только для severity-чипов внутри списка |
| `--radius-input` | 8 px | Синоним `--radius-btn` (инпуты делят токен с кнопками) |
| Spacing | scale 4/8 | Стандартная Tailwind-шкала (gap-2/3/4/6/8/10/12) |

Ширины контейнеров:
- **`max-w-[1200px]`** — основной контейнер шапки, hero, секций и футера.
- **`max-w-[860px]`** — текстовый блок hero (h1 + описание).
- **`max-w-[620px]`** — параграф описания под h1 (ограничение длины строки для читаемости).

Карточки **без теней по умолчанию** — `box-shadow: none`. Hover-эффект — изменение фона, не shadow.

## 8. Адаптивность

Mobile-first. Брейкпоинты Tailwind:

| Брейкпоинт | Ширина | Что меняется |
|------------|--------|--------------|
| base | ≤ 639 px | Одна колонка, кнопки full-width, в шапке скрыта nav и email |
| `sm` | ≥ 640 px | Компактный UI, кнопки auto-width |
| `md` | ≥ 768 px | Грид 2-колонки для карточек преимуществ |
| `lg` | ≥ 1024 px | Полный лейаут, грид 4-колонки, sticky-навигация в шапке отчёта |
| `xl` | ≥ 1280 px | Без изменений, max-width контейнера 1200 px центрированно |

## 9. Структура single-page

Итерация 6 реализует **одну страницу с тремя фазами**, переключаемыми по состоянию скана. Никаких роутов между «главной», «прогрессом» и «результатом» — это один и тот же URL.

### Фаза 0: вход (только для расширенного анализа)

Отдельная страница `/login`, выделенная карточкой по центру вьюпорта. Открывается по клику на «Войти» в шапке. Для базовых проверок этот шаг **не требуется** — анонимный пользователь сразу попадает на Фазу 1.

```
┌──────────────────────────────────────────────────┐
│ Шапка: [Logo + Legal Site]                       │
├──────────────────────────────────────────────────┤
│                                                  │
│            ┌──────────────────────┐              │
│            │                      │              │
│            │ Вход                 │              │
│            │                      │              │
│            │ Логин                │              │
│            │ [.................]  │              │
│            │                      │              │
│            │ Пароль               │              │
│            │ [.................]  │              │
│            │                      │              │
│            │  [    Войти    ]     │  ← primary   │
│            │                      │              │
│            │ Регистрация недоступна — учётная     │
│            │ запись создаётся владельцем сервера. │
│            │                      │              │
│            └──────────────────────┘              │
│                                                  │
│  ← Назад на главную (мелкая ссылка под          │
│     карточкой, для anонимной проверки)           │
└──────────────────────────────────────────────────┘
```

**Параметры формы:**
- Карточка `max-w-[420px]`, фон `bg.base`, граница `1px line`, скругление `radius-card` (16 px), без тени, центрирована по вертикали и горизонтали.
- Поля — `radius-input` (8 px), высота 44 px, `autocomplete="username"` и `autocomplete="current-password"` соответственно. На пароле — без иконки «показать пароль» (MVP, для удешевления; легко добавить позже).
- Кнопка «Войти» — primary, `bg-brand`, белый текст, full-width, `radius-btn`, тот же зелёный, что и CTA «Проверить» на главной.
- Подпись «Регистрация недоступна…» — `ink.secondary`, 13 px, без иконки.
- Ссылка «Назад на главную» под карточкой — `ink.muted`, 13 px, подчёркивание при hover.

**Ошибки:**
- Inline под кнопкой: красный текст `text-sev-crit`, 13 px, мини-иконка `AlertCircle` слева. Формулировки:
  - неверный логин/пароль → «Неверный логин или пароль»;
  - сервер недоступен → «Не удалось войти. Попробуйте ещё раз»;
  - конкретный логин не намекаем, чтобы не давать enumeration-сигнал (даже для одного пользователя).
- Поля остаются заполненными (кроме поля пароля — оно очищается).
- Никаких всплывающих модалок, никаких redirect — пользователь остаётся на `/login`.

**Успех:** редирект на главную (`/`), cookie `legal_site_session` ставит сервер. На главной шапка сразу отрисовывается как «авторизованная» — у фронта при загрузке любой страницы делается `GET /api/v1/auth/me`, по результату определяется состояние шапки.

**Решение «страница vs модалка»** — выбрана отдельная страница `/login`. Причины:
- Чистый URL, можно прямо им поделиться («войди вот тут и попробуй»).
- Проще для статичного экспорта Next.js (`output: 'export'`).
- Модалку усложнило бы взаимодействие с прогрессом/результатами скана.

### Фаза 2: прогресс скана

Hero и блоки главной скрываются. На их месте появляется большая карточка прогресса.

```
┌──────────────────────────────────────────────────┐
│ Шапка                                            │
├──────────────────────────────────────────────────┤
│  ● Проверяем  [E] example.ru        [× Прервать] │
│                                                  │
│  H1: Анализируем cookie и трекеры                │
│  Шаг 5 из 8 · 62%                                │
│                                                  │
│  ▓▓▓▓▓▓▓▓░░░░░░░░  (полоса 8 px, зелёный stripe) │
│                                                  │
│  + Только что: Cookie до получения согласия     ← live-flash
│                                                  │
│  Найдено: [crit 0] [high 2] [med 5] [low 3]     │
│            (серые плитки, цветной шрифт)         │
│                                                  │
│  Этапы:                                          │
│    ✓ Загружаем главную страницу                  │
│    ✓ Ищем политику конфиденциальности            │
│    ⟳ Анализируем cookie и трекеры                │
│    · …                                            │
└──────────────────────────────────────────────────┘
```

Ключевые элементы прогресса:
- **Live-flash** — всплывающее зелёное уведомление `+ Только что: <title нарушения>` при появлении каждого Finding. Анимация 2.6 s (fade-in + hold + fade-out).
- **Favicon домена** в шапке прогресса — цветной кружок с буквой (см. 5.3).
- **Кнопка «Прервать»** в правом верхнем углу — обязательная для `escape-routes` (a11y).

**Шаги скана — НЕ константы UI.** Это группировка нарушений из корпуса по категориям. Источник правды — корпус (`docs/laws/index.yml`) + механизм категоризации, который в [vision.md](vision.md) → «Архитектура» обозначен как `CorpusBundle.for_categories` и реализуется в рамках итерации 6.

Текущее состояние бэкенда (после итерации 4): `app/engine.py` публикует только `scanner_done` + `violation_evaluated` на каждое из 100 нарушений корпуса — концепции «шага» там пока нет.

**Решение по реализации в итерации 6** (один из двух путей):

- **A. Бэк публикует `step`-события.** В `engine.py` добавляется группировка violations по категориям, перед каждой группой emit'ится `ScanEvent(type="step", payload={"name": ..., "index": ..., "total": ...})`. UI просто отображает.
- **B. UI группирует на лету.** Бэк остаётся как есть, UI получает поток `violation_evaluated`, маппит их по `law_id` или `tags` корпуса в категории и сам ведёт счётчик шагов.

Демо-набор шагов в текущем прототипе (`preview.html`) — для визуализации UX и сверки тайминга, **не как контракт**:

1. Загружаем главную страницу
2. Ищем политику конфиденциальности
3. Проверяем формы сбора ПДн
4. Проверяем обязательные сведения
5. Анализируем cookie и трекеры
6. Проверяем рекламные блоки
7. Семантические проверки текстов
8. Финализируем отчёт

В итерации 6 точный список выводится из корпуса (категории + два служебных шага: «Загружаем главную» в начале, «Финализируем отчёт» в конце). Количество и формулировки могут отличаться от 8 выше.

Состояния шага: `·` серый (будущий) → `⟳ Loader2 spin` (текущий) → `✓` зелёный (выполнено).

**Sticky-шапка отчёта** — обязательна. При прокрутке длинного списка нарушений залипшая под основным header'ом полоса (CSS: `position: sticky; top: 64px; backdrop-filter: blur(8px); background: rgba(255,255,255,0.94)`) содержит:
- Favicon домена (32×32 кружок),
- Имя сайта (mono, 15 px, bold),
- Номер аудита (mono, 11 px, в pill-плашке `bg-bg-soft`),
- Мета-строка: «дата, время · N нарушений» (12 px, серая),
- Кнопка **«Новая»** (outline, h-9, иконка `rotate-ccw` + текст; скрыта на mobile до `sm`),
- Кнопка **«PDF»** (зелёная primary, h-9, иконка `file-down` + текст).

Аналогичный `backdrop-filter: blur` стоит на основной шапке страницы (`sticky top-0`).

**Номер аудита** — генерируется на каждый скан, формат `XXXX-MDD` (4 hex-символа в верхнем регистре + номер месяца без ведущего нуля + день с ведущим нулём). Пример: `A7F3-514` для 14 мая. Алгоритм:

```js
const hex = () => Math.floor(Math.random() * 16).toString(16).toUpperCase();
const n = `${hex()}${hex()}${hex()}${hex()}-${new Date().getMonth()+1}${String(new Date().getDate()).padStart(2,'0')}`;
```

**Toolbar над списком нарушений**: слева — счётчик `Всего нарушений: N · требуют ручной проверки: M`, справа — кнопки `Свернуть все` / `Раскрыть все`. Числа подставляются из `ScanResult`.

**Цифры в плитках сводки** окрашены текстовым цветом соответствующего severity (`text-sev-crit` для критических и т. д.), фон плитки — всегда `bg.soft`.

**Финальный CTA-блок** в результатах — повторяет визуальную логику hero-CTA: зелёный фон `#26AD50` + dot-pattern + декоративная SVG-эмблема в углу. Внутри две кнопки:
- Primary «Скачать PDF» — **тёмная** (`bg-ink-primary`), белый текст.
- Secondary «Проверить другой сайт» — **полупрозрачный outline на зелёном** (`bg-white/15` + `border-white/30` + белый текст).

### Карточки нарушений

- **Единый серый фон** `#F4F4F4` для всей карточки (header + body), чёткая граница `1px #ECECEC`, скругление 16 px.
- **Левый стрип 3 px** цвета severity (`inset box-shadow`).
- **Все карточки раскрыты по умолчанию** (требование vision).
- Header: severity-чип + `mono` номер закона + крупный title + chevron.
- Body: описание, блок «Где найдено» с **мини-превью** (отдельная белая карточка `#FFFFFF + border-line`, изображающая фрагмент сайта — НЕ HTML-код, а визуализация: форма без чекбокса, список cookies, footer без ссылки и т. д.). Дальше — два столбца «Штраф» и «Как исправить».

Мини-превью — главное UX-открытие итерации: юристы не понимают HTML, поэтому evidence показывается как «вот так это выглядит на сайте», а не `<form>…</form>`.

**Типы мини-превью — расширяемый набор, зависит от корпуса.** Каждое нарушение в корпусе (`docs/laws/<id>.md` → `violations[].evidence_template` или аналогичное поле, вводится в итерации 6) указывает свой шаблон превью. UI хранит таблицу `шаблон → компонент-визуализатор`. Добавили нарушение в корпус — может потребоваться новый шаблон.

Возможна реализация трёх типов в зависимости от того, как нарушения выводят evidence:

- **Универсальный шаблон** — рендерит generic-карточку: «Где найдено: <text>», моноширинный фрагмент, плашка проблемы. Покрывает большинство нарушений.
- **Специализированные шаблоны** — для часто встречающихся типов: «форма без чекбокса», «список cookies», «футер без ссылки», «HTTP-заголовок» и т. п.
- **Гибрид** — универсальный по умолчанию + специализированные для топ-N типов из корпуса.

**Демо-набор из 6 шаблонов в прототипе** (объект `PREVIEWS` в `preview.html`) — для проверки UX-концепции:

| ID шаблона | Под какое нарушение | Что изображает |
|------------|---------------------|----------------|
| `footer_no_policy` | 152-ФЗ, ст. 18.1 — нет политики ПДн | Футер с 4 ссылками без «Политики обработки…» |
| `form_no_consent` | 152-ФЗ, ст. 9 — нет согласия | Форма обратной связи без чекбокса |
| `cookies_before_consent` | 152-ФЗ + ПП 1119 — cookie до согласия | Список cookies с пометками |
| `contacts_no_requisites` | ЗоЗПП, ст. 9 — нет реквизитов | «Контакты» без ИНН/ОГРН/юр. адреса |
| `banner_no_marking` | 38-ФЗ, ст. 18.1 — нет erid | Рекламный баннер без пометки |
| `dnt_ignored` | Рекомендация DNT | HTTP-запрос с `DNT: 1` |

Шаблонов в итерации 6 может быть меньше (если использовать универсальный) или больше (если каждому нарушению свой). Это решается при реализации.

**Визуальные параметры шаблона** (неизменны):
- Белая карточка `bg.base` + `1px border-line` + `radius 16 px`.
- Внутри: eyebrow-подпись «Что это» + стилизованный фрагмент сайта.
- Цветная плашка проблемы с левой полоской severity-цвета (2 px border-left) + soft-фоном.

**Карточки в marquee «На соответствие чему проверяем»** — **строятся из данных корпуса**:

- **Источник:** `docs/laws/index.yml` (автогенерируется из YAML-фронтматтеров файлов в `docs/laws/`) + одна дополнительная карточка для сводного обзора `koap-overview.md`.
- **Количество карточек** = `len(corpus.laws) + 1`. На момент итерации 5 — 16 (15 актов MVP + КоАП-обзор). При расширении корпуса карусель автоматически вырастет.
- **Ширина** 260 px (mobile) / 280 px (sm+), gap 8 px, `flex flex-col` для одинаковой высоты.
- **Hover:** фон `#EEEEEE`, иконка карточки `scale(1.10)` + фон становится зелёным, цвет иконки белый.

**Поля карточки и их источники:**

| Поле UI | Источник в текущей схеме | Статус |
|---------|--------------------------|--------|
| Название (bold, 15 px) | `short_title` | ✅ есть в YAML |
| Счётчик «N проверок» | `violations_count` | ✅ есть в YAML (для КоАП-обзора показываем «обзор статей») |
| Описание (13 px, secondary) | **нет такого поля** | 🔴 предлагается завести `short_description` в схеме (см. § 15) |
| Иконка Lucide | **нет такого поля** | 🔴 предлагается завести `icon` в схеме (см. § 15) |
| Категория-eyebrow («ПДн», «Реклама» и т. п.) | **нет такого поля** | 🔴 предлагается завести `category` в схеме (см. § 15) |

Сейчас в прототипе три «отсутствующих» поля захардкожены в `LAWS` массиве `preview.html`. В итерации 6 они должны прийти **из корпуса**, чтобы добавление нового закона не требовало правки фронта.

**Блок «Что найдём на сайте» — маркетинговая витрина, НЕ из данных корпуса.** Это сознательное решение (см. § 14): фиксированный список из 6 категорий в коде UI, grid `1 / 2 / 3` колонки (mobile / tablet / desktop) — на десктопе **3 × 2**.

Финальный набор 6 категорий:

| # | Категория | Иконка Lucide | Каким разделам корпуса соответствует |
|---|-----------|---------------|---------------------------------------|
| 1 | Персональные данные | `lock` | 152-ФЗ, 242-ФЗ, ПП 1119 |
| 2 | Cookie и трекеры | `cookie` | 152-ФЗ + ПП 1119, 149-ФЗ |
| 3 | Реклама и маркировка | `megaphone` | 38-ФЗ, ord-ad-marking |
| 4 | Защита потребителей | `users` | ЗоЗПП, ПП 2463, ГК ст. 435–443 |
| 5 | Информация и контент | `file-text` | 149-ФЗ (владелец, контент, хостинг) |
| 6 | Авторское право | `copyright` | ГК ч. IV |

Платежи (161-ФЗ, 54-ФЗ, 63-ФЗ), защита детей (436-ФЗ), государственный язык (53-ФЗ) — в этой витрине **не показываются**, доступны через карусель «На соответствие чему проверяем» ниже. Под блоком — ссылка-якорь: «Платежи, защита детей, гос. язык и другие — в полном списке законов ↓».

При принципиальном расширении корпуса (отраслевые акты — медицина/финансы/азарт) — витрина обновляется ручной правкой, не автоматически. Жёсткий лимит: 6 категорий, чтобы сохранить grid 3 × 2 и читаемость карточек.

### 9.4 Состояния и нестандартные сценарии

Покрывают [vision.md → раздел 5 «Нестандартные ситуации»](vision.md). Все макеты используют ту же палитру и компоненты, что и основная страница.

#### A. Empty state — 0 нарушений

Заменяет блок «Сводка + список нарушений» в результатах. Sticky-bar отчёта (favicon + домен + № аудита + кнопки) остаётся как обычно, под ним правовое уведомление, **затем большая success-карточка** (вместо сводки и списка):

```
┌──────────────────────────────────────────────────┐

│             14 мая 2026, 12:34 · 0 нарушений     │

│ ⓘ Ограничение ответственности. (текст § 13)     │
│                                                  │
│ ┌──────────────────────────────────────────────┐ │
│ │                                              │ │
│ │           [✓ CheckCircle2 64×64,             │ │
│ │             в круге bg-brand-soft]           │ │
│ │                                              │ │
│ │        Нарушений не найдено                  │ │
│ │                                              │ │
│ │   Сайт прошёл все проверки текущего корпуса. │ │
│ │   Это не гарантия полного соответствия —     │ │
│ │   обратите внимание на ограничение           │ │

└──────────────────────────────────────────────────┘
```

PDF выгружается одинаково — это всё ещё валидный отчёт «всё ок».

#### B. Failed state — сайт недоступен (vision сценарии 5.2 / 5.3)

Полная замена результата. Нет шапки отчёта, нет sticky-bar — только карточка с ошибкой по центру.

```
┌──────────────────────────────────────────────────┐
│  Шапка страницы                                  │
├──────────────────────────────────────────────────┤
│                                                  │
│  ┌────────────────────────────────────────────┐  │
│  │  [⚠ AlertTriangle text-sev-high]           │  │
│  │                                            │  │
│  │  Не удалось проверить example.ru           │  │
│  │                                            │  │
│  │  Сайт не отвечает или недоступен.          │  │
│  │  Возможные причины:                        │  │
│  │   · DNS не находит этот домен              │  │
│  │   · Сервер не отвечает в течение 30 секунд │  │
│  │   · Сайт блокирует автоматические проверки │  │
│  │     (Cloudflare, anti-bot защита и т. п.)  │  │
│  │                                            │  │
│  │  [Попробовать снова]  [Указать другой]     │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

Карточка `bg-sev-highSoft` + `border-sev-highBorder` + `radius-card`. Иконка `AlertTriangle` 48×48 цвета `text-sev-high`. Кнопки: «Попробовать снова» — primary (зелёная), «Указать другой» — secondary (outline).

Текст «Возможные причины» подбирается по типу ошибки от бэка (`scanner` payload в SSE-событии `error`):
- DNS error → «DNS не находит этот домен»
- Connection timeout → «Сервер не отвечает в течение 30 секунд»
- 403 / 503 Cloudflare-сигнатура → «Сайт блокирует автоматические проверки»
- Любое другое → general fallback с тремя пунктами.

#### C. Queued state — семафор занят (vision сценарий 3)

Появляется вместо обычной шапки прогресса, если идёт другой скан (`asyncio.Semaphore(1)` в бэке). После освобождения семафора плавно переходит в обычную прогресс-секцию (§ 9, Фаза 2).

```
┌──────────────────────────────────────────────────┐
│ ●  В очереди                                     │
│                                                  │
│ Проверяем  [E] example.ru                        │
│                                                  │
│ Сейчас выполняется другая проверка. Ваша         │
│ начнётся автоматически, как только предыдущая    │
│ завершится. Обычно — через 1–3 минуты.           │
│                                                  │
│ ▰▰▰▰▰▰▰▰▰▰  (индетерминированная зелёная          │
│              полоса, без процентов)              │
│                                                  │
│ Можно закрыть вкладку — проверка продолжит       │
│ работать. По возврату на сайт в течение часа     │
│ результат будет доступен.                        │
│                                                  │
└──────────────────────────────────────────────────┘
```

Отличия от обычной прогресс-карточки:
- Заголовок «В очереди» вместо «Проверяем».
- Прогресс-бар индетерминированный (`@keyframes indeterminate` — слайд `0 → 100%` ширины 30 px на серой подложке).
- Нет списка шагов и счётчиков severity.
- Внизу — пояснение про закрытие вкладки.

#### D. UX-flow кнопки «Прервать»

Не отдельный экран — последовательность действий:

1. Клик на «× Прервать» в правом верхнем углу шапки прогресса.
2. **Confirm-диалог** (shadcn `AlertDialog`): «Прервать проверку example.ru? Прогресс не сохранится» + кнопки «Прервать» (destructive, цвет `text-sev-crit`) и «Продолжить» (default).
3. При подтверждении: `DELETE /api/v1/scans/{id}` (контракт детализируется в итерации 6) → возврат на главную → форма URL очищена → **toast** в правом нижнем углу: «Проверка прервана» (auto-dismiss 4 s, `aria-live="polite"`).
4. При отмене Confirm — остаёмся в прогрессе.

## 10. Анимации

Все длительности — в диапазоне 150–400 ms, easing `ease-out` / `cubic-bezier(0.16, 1, 0.3, 1)`. Никаких bouncy-overshoot.

| Элемент | Длительность | Эффект |
|---------|-------------|--------|
| Hero rise (h1, описание, форма) | 360 ms ease-out, stagger 40 ms | `translateY(6px) → 0` + `opacity 0 → 1` |
| Появление прогресс-секции | 440 ms | `translateY(16px) → 0` + fade |
| Прогресс-бар (заливка) | 600 ms cubic-bezier | width change |
| Прогресс-бар (stripe) | 1 s бесконечно | анимированная диагональная штриховка |
| Pulse-dot (зелёная точка «Проверяем») | 1.4 s бесконечно | scale + opacity |
| Spin (Loader2 у текущего шага) | 1.1 s бесконечно | rotate |
| Count bump (счётчики при инкременте) | 320 ms | scale 1 → 1.06 → 1 |
| Live-flash (всплывающее уведомление) | 2.6 s | fade-in + hold + fade-out |
| Marquee (карусель законов) | 55 s linear (40 s на mobile ≤ 640 px) бесконечно | translateX, hover-pause |
| Hover карточек | 200 ms | background / scale иконки |
| Раскрытие/сворачивание нарушения | 280–320 ms | `max-height` + opacity |

`@media (prefers-reduced-motion: reduce)` — обязательно. Отключает marquee, shimmer-stripe, pulse-dot, оставляет переходы статической смены состояний.

## 11. Доступность (a11y) — минимум для MVP

- Контраст основного текста на белом ≥ 4.5:1 (`#1F1F1F` на `#FFFFFF` = ~15:1, проходит AAA).
- Severity-различия дублируются текстом и иконкой, не только цветом (правило `color-not-only`).
- **Skip-link «Перейти к содержимому»** в самом начале body — для клавиатурной навигации.
- **Hidden labels** (`<label class="sr-only">`) для URL-инпута + `autocomplete="url"` + `inputmode="url"` (триггерит правильную мобильную клавиатуру).
- **`aria-label`** на icon-only кнопках (Выйти, Прервать).
- **`aria-live="polite"`** + **`aria-busy`** на прогресс-секции — screen reader озвучивает смену шагов.
- **`prefers-reduced-motion`** соблюдается.
- **`touch-action: manipulation`** на интерактивных элементах — убирает 300 ms tap-delay на мобильных.
- Высота touch-целей ≥ 40 px (правило `touch-target-size`).
- Видимое фокус-кольцо `:focus-visible` 2 px зелёного цвета.

## 12. Логотип и брендинг

- **Название продукта в UI:** `Legal Site` (с пробелом, оба слова с заглавной). В путях файлов и репозитории остаётся `Legal_site` как технический id.
- **Логотип-знак:** SVG-щит `#26AD50` с белой галочкой внутри, размер 28×28 px в шапке, 22×22 px в футере. Inline в HTML (без внешних файлов на старте).
- **Иконка в шапке + текст «Legal Site»** идут одной строкой без подчёркивания.

## 13. Юридическая формулировка «Ограничение ответственности»

Полный текст плашки в результатах (закреплён, при изменении правится только здесь):

> **Ограничение ответственности.** Отчёт сформирован автоматизированным сканером, носит предварительный информационно-справочный характер и не является юридическим заключением. Применимость отдельных норм к конкретному сайту зависит от обстоятельств, которые автоматическая проверка не учитывает в полном объёме (организационно-правовая форма владельца, состав обрабатываемых данных, особенности бизнес-процессов и т. п.). Перед принятием решений на основании отчёта рекомендуется согласовать выводы с практикующим юристом.

Сокращённый дубль — в футере мелким шрифтом:

> Отчёт носит информационно-справочный характер и не заменяет юридическое заключение. Для принятия решений на основании отчёта согласуйте выводы с практикующим юристом. Результаты проверок не сохраняются.

Размещение: плашка с иконкой `info` (`#1F6FBF`) на бледно-сером фоне — **под sticky-шапкой отчёта, до сводки**. Тёмно-цветной фон у плашки не использовать — это не error/warning, а информационная справка.

## 14. Зависимости от корпуса законов (что НЕ зашито в UI)

UI — **витрина** над корпусом и движком, а не источник правды о праве. Следующие элементы дизайна привязаны к данным, а не к константам:

| Элемент UI | Источник данных | Что произойдёт при расширении корпуса |
|------------|-----------------|---------------------------------------|
| Карточки в marquee (название, описание, иконка, категория, число проверок) | `docs/laws/index.yml` + поля `short_title`, `short_description`, `icon`, `category`, `violations_count` (последние 3 — после расширения схемы в итерации 6, см. § 15) | Карточек станет больше, дизайн карточки не меняется |
| Счётчик «15 актов · 100 проверок» | `len(corpus.laws)`, `sum(law.violations_count)` | Цифры обновятся автоматически |
| Категории «Что найдём на сайте» (топ-6) | **Маркетинговая витрина** — не из корпуса; фиксированный список 6 категорий в коде UI. См. § 9, Фаза 1 | Меняется только ручной правкой при принципиальном расширении корпуса (новые отрасли) |
| Шаги прогресса скана | `CorpusBundle.for_categories` (итерация 6) или категоризация на стороне UI по полю `category` | Число и формулировки шагов меняются |
| Текст «Только что: …» в live-flash | `Finding.title` из бэка | Привязан к нарушению в корпусе |
| Типы мини-превью evidence | Поле `evidence_template` нарушения в корпусе (вводится в итерации 6) | Может потребоваться добавить компонент-визуализатор |
| Severity-чипы и палитра severity | `Violation.severity` (`low`/`medium`/`high`/`critical`) | Палитра не меняется; новые severity-уровни — отдельный ADR |
| Карточка нарушения (header + тело) | `Finding` целиком | Структура полей зафиксирована, новые поля — отдельный ADR |
| Состояние шапки (Войти / `<login>` + Выйти) и тоггл «Расширенный анализ» | `GET /api/v1/auth/me` при загрузке любой страницы (cookie `legal_site_session`) | Без cookie — анонимная ветка; с валидной сессией — авторизованная; logout очищает cookie и переключает шапку |

**Что в UI действительно зашито как константа (не зависит от корпуса):**

- Базовая палитра (фоны, текст, бренд, severity-цвета).
- Типографика (Onest + JetBrains Mono, шкала размеров).
- Скругления, spacing, брейкпоинты.
- Структура single-page (3 фазы) и их компоновка.
- Анимации, длительности, easing.
- Текст «Ограничение ответственности» (§ 13).
- Логотип, название продукта.
- A11y-минимум (§ 11).

## 15. Унаследованные задачи для итерации 6

Дизайн-итерация закрывается с двумя задачами, которые **не относятся к дизайну**, но должны быть решены в начале имплементации фронта. Зафиксированы здесь и продублированы в карточке итерации 6 в [plan.md](plan.md) как «Предусловия».

1. **ADR-0002 — расширение схемы корпуса полями для UI.** Правка [docs/laws/schema.md](laws/schema.md) + 15 YAML-файлов корпуса + пересборка `docs/laws/index.yml` через `python tools/rebuild_index.py`. Новые поля:
   - `short_description: "О персональных данных"` — короткая суть закона (1 строка, ≤ 60 символов) для карточки в marquee.
   - `icon: lock` — имя иконки из набора Lucide.
   - `category: privacy` — одна из 6 категорий витрины (`privacy`, `cookies`, `advertising`, `consumer`, `info`, `copyright`). Поле опциональное: оно нужно для маппинга «закон → категория», но саму витрину «Что найдём» это поле не определяет (она зафиксирована, см. § 9, Фаза 1).
   - `evidence_template: form_no_consent` — на уровне `violations[]`, имя шаблона мини-превью.

   До этого ADR на UI работает хардкод-маппинг `id → описание/иконка/категория` в массиве `LAWS` прототипа.

2. **`CorpusBundle.for_categories` — категоризация нарушений для шагов прогресса.** Архитектурное решение: бэк публикует `step`-события (вариант A в § 9, Фаза 2) или UI группирует `violation_evaluated` на лету по полю `category` (вариант B). От этого зависит контракт SSE-стрима в [app/events.py](app/events.py) и набор шагов в UI. Категории берутся из `category` каждого закона (см. ADR-0002).
