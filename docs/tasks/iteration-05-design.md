# Tasklist — Итерация 5: Выбор и согласование дизайна UI

Связь с обзором: [docs/plan.md](../plan.md), карточка «Итерация 5».

**Особенность итерации:** не пишем код — собираем дизайн-документ и получаем явное согласование пользователя. Артефакт на выходе — [docs/design.md](../design.md). Итерация закрывается только после фразы пользователя «согласовано».

## Этапы

### 1. Стартовый раунд вопросов

Снять с пользователя ключевые ориентиры до того, как тратить силы на мокапы:

- Референс-вектор (3–4 готовых варианта от Claude с пояснением — у пользователя своих референсов нет).
- Тональность (деловой/доверительный vs дружелюбный vs сбалансированный).
- UI-стек (чистый Tailwind vs Tailwind + надстройка).
- Адаптивность (desktop / desktop+tablet / mobile-first).

### 2. Подобрать референсы

Claude предлагает 3–4 кандидата с короткой характеристикой (палитра, плотность, тональность). Пользователь выбирает направление (или комбинирует). Ссылки и пояснения сохраняем в `docs/design.md`.

### 3. Зафиксировать стек

Записать в `docs/design.md` решение «чистый Tailwind / + shadcn/ui / + Headless UI / + Radix» с аргументацией одной строкой (вписывается в принцип «без избыточных абстракций» из vision).

### 4. Палитра

- Основной фон, поверхность карточек, текст основной/вторичный.
- Акцентный цвет (брендовый).
- Severity-цвета для нарушений: `low` / `medium` / `high` / `critical` — должны различаться при дальтонизме (не только hue, но и насыщенность/иконка).
- Цвет состояния `inconclusive` (отдельная секция в результате — нужна нейтральная палитра).

### 5. Типографика

- Один шрифт основного текста (sans-serif, с кириллицей).
- Один моноширинный для evidence-цитат (селекторы, фрагменты HTML).
- Базовая шкала размеров (h1/h2/h3/body/caption).

### 6. Адаптивность

Зафиксировать решение из этапа 1 и описать поведение на выбранных размерах (брейкпоинты в терминах Tailwind, если решено поддерживать ≥ tablet).

### 7. ASCII-мокапы трёх экранов

- **Главная** — инпут URL + кнопка «Проверить», описание выше/ниже.
- **Прогресс** — текущий шаг (название проверки), счётчики найденных нарушений по severity, поведение при «ждём предыдущий скан» (см. vision, сценарий 3).
- **Результат** — список карточек Finding (все раскрыты по умолчанию), кнопки «Свернуть все» / «Раскрыть все», секция «inconclusive», кнопка «Скачать PDF», шапка со сводкой.

Каждый мокап — текстовый/ASCII блок в `docs/design.md` + 1–2 абзаца с объяснением логики.

### 8. Сборка `docs/design.md`

Структура документа:

1. Контекст и аудитория (две-три строки из idea.md).
2. Выбранный референс-вектор + ссылки/описание.
3. Тональность и принципы языка.
4. Стек (Tailwind + надстройка / без).
5. Палитра (таблица hex-кодов с пояснениями + severity-карта).
6. Типографика.
7. Адаптивность.
8. ASCII-мокапы трёх экранов.
9. Открытые вопросы (если останутся — выносим в задел итерации 6).

ADR ([docs/adr/0002-ui-design.md](../adr/0002-ui-design.md)) заводим **только если** в процессе всплывут решения архитектурно значимого уровня (например, отказ от static export или нестандартная схема сборки). Иначе ограничиваемся `docs/design.md`.

### 9. Запрос явного согласования

После сборки документа — отдельным сообщением: «Готов финальный `docs/design.md` — подтверди, что согласовано, или вернёмся править». Пока пользователь не сказал «согласовано» — итерация не закрывается.

### 10. Закрытие итерации

- Статус в [plan.md](../plan.md) `🚧 → ✅`.
- Финальный коммит — черновик сообщения показать пользователю до `git commit`.
- Демонстрация: ссылка на `docs/design.md` + указатель на ключевой раздел (мокапы).

## Принципы при работе с этой итерацией

- Не лезем во фронтенд-код — только текстовые артефакты.
- Не плодим референс-картинки в репозитории, если можно обойтись ссылками + описанием (папка `docs/design/` создаётся только если пользователь принесёт собственные скриншоты).
- Все решения формулируем одной–двумя строками с аргументацией — `docs/design.md` должен читаться за 5 минут, а не за полчаса.
