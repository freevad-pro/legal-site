# Границы верификации данных в справочнике

## Зачем этот документ

Корпус законов в `docs/laws/` собран по официальным источникам (pravo.gov.ru,
КонсультантПлюс, Гарант) и сайтам регуляторов (Роскомнадзор, ФАС,
Роспотребнадзор, ФНС). При этом часть данных **не получена напрямую**
с pravo.gov.ru (платные источники требуют подписки), а часть нормативной
информации **объективно не публикуется в открытом виде**.

Этот файл фиксирует **все известные границы верификации** в одном месте,
с привязкой к конкретным нарушениям, и описывает, **как с этим работать**:
- людям, читающим справочник, — где НЕ полагаться на цифру дословно;
- движку проверки сайтов — где автоматически добавить оговорку в отчёт.

## Что записано

Блок `caveats:` в YAML-фронтматтере — машиночитаемый перечень. Каждая запись:
- `id` — стабильный идентификатор,
- `affects_law_ids` / `affects_violation_ids` — к каким объектам корпуса применяется,
- `affects_subjects` / `affects_articles` — точная зона действия (опционально),
- `severity` — `low` / `medium` (на риск принятия решения),
- `message` — содержательная оговорка для пользователя,
- `user_advice` — что конкретно делать,
- `sources_to_confirm` — куда отправить за подтверждением.

## Инструкция для движка проверки сайтов

При формировании отчёта по сайту:

1. **Найди нарушения**, обнаруженные на сайте, и определи их `violation_id`.
2. **Сопоставь с `caveats[]`** в этом файле: если `affects_violation_ids`
   содержит данный `violation_id` (или значение `all` / `many`), оговорка
   применима.
3. **При `affects_subjects`** дополнительно сверь субъекта проверки
   (организация / ИП / физлицо). Применяй оговорку только если субъект
   попадает в список.
4. **Выведи в отчёт** блок «Замечания о точности данных» с содержимым
   `message` и `user_advice`, со ссылками из `sources_to_confirm`.
5. **Severity влияет на оформление**: `medium` — заметная пометка рядом
   с суммой штрафа; `low` — сворачиваемая сноска внизу карточки нарушения.

Псевдокод:

```python
def render_violation(violation, subject):
    out = base_render(violation, subject)
    for caveat in coverage_limitations.caveats:
        if not violation_matches(violation.id, caveat.affects_violation_ids):
            continue
        if caveat.affects_subjects and subject not in caveat.affects_subjects:
            continue
        out.add_caveat(
            severity=caveat.severity,
            text=caveat.message,
            advice=caveat.user_advice,
            sources=caveat.sources_to_confirm,
        )
    return out
```

## Принципы

- **Не скрывать ограничения.** Если данные оценочные — пользователь должен
  это знать в момент чтения отчёта, а не после проверки у юриста.
- **Указывать, куда идти.** Каждая оговорка содержит `sources_to_confirm` —
  не оставлять пользователя с пустыми руками.
- **Привязка к месту в корпусе.** Оговорки относятся к конкретным
  `violation_id` / `law_id`, не «вообще к справочнику» — иначе они
  превратятся в шум.
- **Поддержка в актуальном состоянии.** Когда данные удаётся подтвердить
  (платная подписка, прямой запрос регулятора) — соответствующий
  `caveat` удаляется из перечня, в `review_log` соответствующего файла
  фиксируется источник верификации.

## Как добавить новую оговорку

1. Добавь запись в `caveats[]` этого файла. Используй стабильный `id`
   в kebab-case.
2. В затронутом файле (`docs/laws/...`) при необходимости обнови
   `verification_notes` — кратко упомяни оговорку, чтобы при чтении
   YAML фронтматтера было видно.
3. Запусти `python tools/rebuild_index.py`. Скрипт не валидирует
   `caveats[]` напрямую (это compendium, не закон), но проверит общую
   целостность.
4. После того как движок начнёт читать `caveats[]` — добавь
   unit-тесты на пути показа.
