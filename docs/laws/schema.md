# Схема справочника законов

Этот документ описывает формат файлов в `docs/laws/`. Каждый закон / нормативный акт = один Markdown-файл с YAML-фронтматтером. Фронтматтер парсится движком проверки, тело Markdown читают люди.

## Структура файла

```markdown
---
{YAML-фронтматтер — обязательные и опциональные поля, см. ниже}
---

# Заголовок закона

## Кратко
## Кому применяется
## Ключевые требования
## Что должно быть на сайте
## Изменения 2022–2026
## Связанные акты
```

## Поля YAML-фронтматтера

### Идентификация (обязательно)

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string (kebab-case) | Стабильный идентификатор: `152-fz`, `pp-1119`, `gk-rf-part-iv`. Не зависит от редакции. |
| `title` | string | Полное название акта в кавычках. |
| `short_title` | string | Короткое название для UI и ссылок: `"152-ФЗ"`. |
| `type` | enum | `federal_law` \| `code` \| `rf_law` \| `gov_decree` \| `order` \| `regulation` |
| `number` | string | Номер с буквенной частью: `"152-ФЗ"`, `"2300-1"`, `"№ 1119"`. |

### Даты (обязательно)

Все даты — в ISO 8601 (`YYYY-MM-DD`). Если точная дата неизвестна — указать первый день месяца и пометить `verified: partial`.

| Поле | Описание |
|------|----------|
| `adopted_date` | Дата принятия. |
| `in_force_since` | Дата вступления в силу первой редакции. |
| `last_amended` | Дата последней значимой редакции, релевантной для сайтов. |

### Статус (обязательно)

| Поле | Значения |
|------|----------|
| `status` | `in_force` \| `repealed` \| `not_yet_in_force` |

### UI-метаданные (обязательно)

Поля для отображения в карусели законов и группировки в шагах прогресса (см. [ADR-0002](../adr/0002-corpus-ui-fields.md)).

| Поле | Тип | Описание |
|------|-----|----------|
| `category` | enum | Одна из: `privacy`, `cookies`, `advertising`, `consumer`, `info`, `copyright`. Фронт группирует нарушения по этой категории в шагах прогресса. |
| `icon` | string | Имя иконки из набора Lucide в kebab-case: `lock`, `database`, `megaphone`, `tag`, `users`, `shopping-cart`, `scroll`, `receipt`, `credit-card`, `languages`, `file-text`, `shield`, `pen-tool`, `copyright`, `shield-check`. |
| `short_description` | string (≤ 60 символов) | Короткая суть закона для карточки в карусели на главной (одна строка). Не дублирует `short_title`. Пример: `"О персональных данных"`, `"Правила продажи товаров"`. |

```yaml
category: privacy
icon: lock
short_description: "О персональных данных"
```

### Источники (обязательно, ≥ 1)

```yaml
official_sources:
  - title: "Официальный интернет-портал правовой информации"
    url: https://pravo.gov.ru/...
  - title: "КонсультантПлюс"
    url: https://www.consultant.ru/...
```

**Порядок приоритета:** pravo.gov.ru / publication.pravo.gov.ru → КонсультантПлюс → Гарант → сайт регулятора.

### Регуляторы и охват

```yaml
regulators:                  # органы, контролирующие исполнение
  - Роскомнадзор
  - ФАС
applies_to:                  # категории сайтов, к которым применим
  - all_websites             # любой сайт
  # допустимы также: ecommerce | landing | blog | media | service | gov | b2b
```

**Важно:** значение `all_websites` — взаимоисключающее с остальными. Если закон применим к любому сайту, в списке должно быть **только** `all_websites`. Конкретные категории (`ecommerce`, `landing` и т. п.) используются, когда закон применим лишь к подмножеству сайтов.

### Связи и теги

```yaml
related: [242-fz, pp-1119]            # id законов из docs/laws/, существующих как файлы
references_in_common: [koap-overview] # ссылки на сводные обзоры в docs/laws/common/
tags: [personal_data, consent, cookies]
```

**`related`** ссылается только на **самостоятельные акты** в `docs/laws/`. Поправки в текущий закон (266-ФЗ, 420-ФЗ и т. п.) НЕ выносим в `related` — они описываются внутри файла в разделе «Изменения». Сводные обзоры (`common/koap-overview.md` и др.) ссылаются через `references_in_common`.

### Свежесть данных (обязательно)

```yaml
verified_at: 2026-05-13      # ISO 8601, когда последний раз сверяли с источниками
verified_by: agent-152fz     # кто/что проверял: agent-<id> или human-<email>
verified: partial            # full | partial — partial = есть пробелы/TODO в файле
verification_notes:          # обязательно при verified: partial — список открытых вопросов
  - "Уточнить часть ст. 13.11 КоАП для трансграничной передачи (ч. 1.1 vs ч. 8) по pravo.gov.ru"
  - "Проверить порог оборотного штрафа: 20 vs 25 млн руб"
```

**Правило:** если хотя бы один факт в файле не подтверждён двумя источниками или есть спорные нумерации статей — `verified: partial` + перечислить открытые вопросы в `verification_notes`. Поле `full` ставится только когда вообще нет TODO.

### Нарушения (обязательно, ≥ 1)

Это главный блок — именно его потребляет движок проверки.

```yaml
violations:
  - id: 152-fz-no-consent-form        # глобально уникальный id нарушения
    article: "ст. 9, ч. 1"            # статья нарушаемого закона
    title: "Сбор ПДн без явного согласия пользователя"
    severity: high                    # low | medium | high | critical
    description: |
      Подробное описание нарушения для отчёта пользователю (1–3 предложения).

    detection:                        # машиночитаемые сигналы для движка
      page_signals:                   # признаки на уровне отдельной страницы
        - type: form_without_consent_checkbox
          description: "Форма содержит поля для ПДн, но рядом нет чекбокса согласия"
          html_patterns:
            - 'input[type="email"]'
            - 'input[name*="phone"]'
            - 'input[name*="name"]'
          required_absent:
            - "checkbox связанный с согласием на обработку ПДн"
      site_signals:                   # признаки на уровне всего сайта
        - type: privacy_policy_missing
          description: "Отсутствует страница политики обработки ПДн"
          check: lookup_pages_by_keywords
          keywords:
            - "политика конфиденциальности"
            - "обработка персональных данных"
            - "privacy policy"

    penalties:                        # штрафы по КоАП или иным актам
      - subject: citizen              # citizen | official | sole_proprietor | small_org | organization
        coap_article: "ст. 13.11 КоАП РФ, ч. 1"
        amount_min: 6000
        amount_max: 10000
        currency: RUB
        notes: "За первичное нарушение"
      - subject: organization
        coap_article: "ст. 13.11 КоАП РФ, ч. 1"
        amount_min: 60000
        amount_max: 100000
        currency: RUB

    recommendation: |
      Что должен сделать владелец сайта, чтобы устранить нарушение.
      Будет показано в отчёте.

    references:
      - "https://rkn.gov.ru/..."     # ссылки на разъяснения регулятора

    evidence_template: form_no_consent   # опционально; см. ниже
```

**`evidence_template`** (опционально) — имя шаблона мини-превью на странице результата. Закрытый словарь (синхронизирован с React-компонентами `frontend/src/components/evidence/*.tsx`):

- `footer_no_policy` — отсутствует ссылка на политику конфиденциальности в подвале.
- `form_no_consent` — форма ввода ПДн без чекбокса согласия.
- `cookies_before_consent` — cookies устанавливаются до получения согласия.
- `contacts_no_requisites` — нет реквизитов продавца / контактных данных.
- `banner_no_marking` — рекламный баннер без маркировки `ERID`.
- `dnt_ignored` — игнорируется заголовок `DNT` / явный отказ от трекинга.

Если поле не задано — UI рендерит универсальный `GenericEvidence` (mono-цитата + severity-чип). Введение нового значения требует одновременной правки [ADR-0002](../adr/0002-corpus-ui-fields.md), схемы и фронта.

### Требования к движку проверки (на что опираются `detection`)

- **HTML-паттерны** в `html_patterns` — это **CSS-селекторы**. Допустим синтаксис CSS Selectors Level 4, включая `[attr*="value" i]` для case-insensitive matching. Движок должен использовать парсер с поддержкой L4 (например, BeautifulSoup ≥ 4.7 с `soupsieve`, или браузерный `document.querySelectorAll`).
- **`required_absent`** — список селекторов или описаний; нарушение считается обнаруженным, если на странице **отсутствуют** все указанные элементы.
- **`required_keywords`** — слова, которые должны присутствовать в указанном источнике (тексте политики, заголовке и т. п.). Поиск без учёта регистра и с нормализацией пробелов.
- **`check`** — название функции движка, которая выполняет сложную проверку, не сводимую к селекторам (`lookup_pages_by_keywords`, `http_status_check`, `rkn_registry_lookup`, `ip_geolocation`, `form_action_geo`, `cookie_set_before_consent`, `text_length_threshold`, `date_in_document`, `api_endpoint_scan`, `link_near_form_to_privacy`). Новые `check` добавляются в реестр функций движка при добавлении нарушений.

### Словари допустимых значений

**`severity`** — насколько критично нарушение:
- `low` — формальное замечание, штраф маловероятен или минимален.
- `medium` — точно есть штраф, но небольшой.
- `high` — серьёзные штрафы, частые проверки регулятора.
- `critical` — оборотные штрафы, блокировка сайта, уголовная ответственность.

**`subject`** — субъект ответственности:
- `citizen` — гражданин (физлицо).
- `official` — должностное лицо.
- `sole_proprietor` — индивидуальный предприниматель.
- `small_org` — субъект малого предпринимательства (ст. 4.1.2 КоАП — половинный штраф).
- `organization` — юридическое лицо.

**`applies_to`** — категории сайтов:
- `all_websites` — любой сайт.
- `ecommerce` — интернет-магазины, торговля.
- `landing` — лендинги, страницы продуктов.
- `blog` — блоги, медиа-проекты.
- `media` — СМИ, новостные сайты.
- `service` — сайты услуг.
- `gov` — государственные/муниципальные.
- `b2b` — корпоративные.

**`type`** — тип нормативного акта:
- `federal_law` — Федеральный закон (ФЗ).
- `code` — Кодекс (ГК РФ, КоАП РФ, НК РФ).
- `rf_law` — Закон РФ старого образца (например, 2300-1).
- `gov_decree` — Постановление Правительства РФ (ПП РФ).
- `order` — Приказ ведомства.
- `regulation` — Регламент / Положение.

**`category`** — тематическая группа закона для UI:
- `privacy` — обработка персональных данных, локализация, защита ПДн.
- `cookies` — cookie-баннеры, трекинг, согласия на cookies (резерв; в MVP-корпусе не используется).
- `advertising` — реклама, маркировка интернет-рекламы.
- `consumer` — права потребителей, правила продаж, оферта, расчёты.
- `info` — общие требования к информации на сайте, госязык, защита детей, электронная подпись.
- `copyright` — интеллектуальная собственность, авторские права.

## Структура тела Markdown

После закрывающих `---` пишем человекочитаемое описание. Рекомендованные разделы:

```markdown
# Полное название закона

## Кратко
2–3 абзаца: что регулирует, кого касается, почему важно для сайтов.

## Кому применяется
Какие категории сайтов / операторов попадают под регулирование. Исключения.

## Ключевые требования
Нумерованный список с конкретными статьями.

## Что должно быть на сайте
Практический чек-лист — то же, что в блоках `detection`, но человеческим языком.

## Изменения 2022–2026
Значимые редакции с датами и сутью изменения.

## Связанные акты
Ссылки на другие файлы справочника через `[[id]]` (например, `[[242-fz]]`).
```

## Соглашения по именованию файлов

- Имя файла = `{id}-{slug}.md`, всё в kebab-case.
- Для подзаконных актов префикс `pp-` (постановление правительства), `prikaz-` (приказ).
- Примеры: `152-fz-personal-data.md`, `pp-1119-pdn-protection.md`, `gk-rf-part-iv-copyright.md`.

## Как обновлять справочник

1. **Изменилась редакция закона:** правим только соответствующий файл, обновляем `last_amended` и `verified_at`, перегенерируем `index.yml`.
2. **Появилось новое нарушение / детектор:** добавляем элемент в `violations:` в файле закона.
3. **Появился новый закон:**
   - Создаём файл `docs/laws/{id}-{slug}.md` по этой схеме.
   - Добавляем запись в `index.yml`.
   - Добавляем строку в `README.md`.
4. **Закон отменён:** меняем `status: repealed`, оставляем файл (для исторических ссылок), убираем из `applies_to` в `index.yml`.

## Целостность

- Все `id` в `related: [...]` должны существовать как файлы в `docs/laws/`.
- Все ссылки `[[id]]` в теле тоже должны резолвиться.
- Все `id` нарушений (`violations[].id`) глобально уникальны во всём справочнике.
- `verified_at` старше 6 месяцев — повод перепроверить актуальность.
