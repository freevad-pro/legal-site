# ADR-0002 — Расширение схемы корпуса полями для UI

- **Статус:** Accepted
- **Дата:** 2026-05-14
- **Автор:** vadimwork011@gmail.com (совместно с Claude)
- **Связанные документы:** [docs/adr/0001-laws-reference-structure.md](0001-laws-reference-structure.md), [docs/laws/schema.md](../laws/schema.md), [docs/design.md §15](../design.md), [docs/tasks/iteration-06-frontend.md](../tasks/iteration-06-frontend.md)

## Контекст

К итерации 6 (Frontend MVP) дизайн ([docs/design.md](../design.md)) задал три новых элемента, которые требуют от корпуса законов структурированных метаданных:

1. **Карусель «16 законов»** на главной — каждый закон рисуется карточкой с собственной иконкой Lucide и короткой подписью (1 строка, ≤ 60 символов).
2. **Шаги прогресса скана** — нарушения группируются по тематике (privacy / advertising / consumer / info / copyright). Группа — это category, а не tag: тегов в корпусе уже 50+ и они не образуют дисциплинированной системы.
3. **Мини-превью нарушений** на странице результата — для 6 типовых сюжетов из прототипа в карточке нарушения отрисовывается стилизованный фрагмент сайта (форма без согласия, баннер cookies до согласия, реквизиты продавца, и т. п.). Для остальных нарушений работает универсальный fallback.

Сейчас корпус ([docs/laws/schema.md](../laws/schema.md)) этих метаданных не содержит. Пока их нет, UI вынужден держать хардкод-маппинг `law_id → {icon, category, short_description}` и `violation_id → evidence_template` прямо в коде фронта — это дублирует знание и расходится с принципом «корпус — единственный источник правды о праве», заданным в [ADR-0001](0001-laws-reference-structure.md).

Отдельно — решение о механизме шагов прогресса в SSE-потоке (см. [docs/design.md §15](../design.md), пункт 2): либо бэк публикует `step_started`/`step_done`-события (вариант A), либо UI группирует приходящие `violation_evaluated` на лету по `category` закона (вариант B). Это влияет на расширение схемы: при варианте B `category` достаточно на уровне `Law`; при варианте A пришлось бы добавлять её на уровне `Violation` (или делать промежуточный группировщик в engine).

## Решение

В YAML-фронтматтер корпуса добавляются **три новых обязательных поля** на уровне закона и **одно опциональное** на уровне нарушения. Дополнительно фиксируется выбор варианта B для SSE-прогресса.

### 1. Новые поля

**На уровне закона (`Law`) — обязательные:**

| Поле | Тип | Описание |
|------|-----|----------|
| `category` | enum | Одна из: `privacy`, `cookies`, `advertising`, `consumer`, `info`, `copyright`. Используется фронтом для группировки нарушений по шагам прогресса и для маппинга в карточки витрины «Что найдём». |
| `icon` | string (Lucide kebab-имя) | Имя иконки из набора Lucide, например `lock`, `database`, `megaphone`. Отрисовывается в карусели законов на главной и в шапке секций отчёта. |
| `short_description` | string (≤ 60 символов) | Короткая суть закона для карточки в карусели (одна строка). Не дублирует `short_title` — это не «152-ФЗ», а «О персональных данных». |

**На уровне нарушения (`Violation`) — опциональное:**

| Поле | Тип | Описание |
|------|-----|----------|
| `evidence_template` | string \| null | Имя шаблона мини-превью на странице результата. Допустимые значения: `footer_no_policy`, `form_no_consent`, `cookies_before_consent`, `contacts_no_requisites`, `banner_no_marking`, `dnt_ignored`. При отсутствии (`null` / поле не задано) UI рендерит универсальный `GenericEvidence` (mono-цитата + severity-чип). |

### 2. Перечисление категорий

Enum `LawCategory` зафиксирован в формальном словаре схемы:

- `privacy` — обработка персональных данных, локализация, защита ПДн.
- `cookies` — cookie-баннеры, трекинг, согласия на cookies. В корпусе MVP отдельных «cookie-законов» нет (тема покрыта 152-ФЗ и сводным обзором [common/cookies-in-russia.md](../laws/common/cookies-in-russia.md)), но enum резервирует значение — карточка «Cookies» на главной всегда отрисовывается, а в будущем сюда может попасть отдельный акт.
- `advertising` — реклама, маркировка интернет-рекламы.
- `consumer` — права потребителей, правила продаж, оферта, расчёты.
- `info` — общие требования к информации на сайте, госязык, защита детей, электронная подпись.
- `copyright` — интеллектуальная собственность, авторские права.

В MVP-корпусе из 15 законов используются 5 категорий: privacy (3), advertising (2), consumer (5), info (4), copyright (1). Маппинг по файлам зафиксирован в [docs/tasks/iteration-06-frontend.md](../tasks/iteration-06-frontend.md).

### 3. Перечисление шаблонов превью

`evidence_template` имеет закрытый словарь (6 значений), синхронизированный с 6 React-компонентами в `frontend/src/components/evidence/`:

- `footer_no_policy` — отсутствует ссылка на политику конфиденциальности в подвале.
- `form_no_consent` — форма ввода ПДн без чекбокса согласия.
- `cookies_before_consent` — cookies устанавливаются до получения согласия.
- `contacts_no_requisites` — нет реквизитов продавца / контактных данных.
- `banner_no_marking` — рекламный баннер без маркировки `ERID`.
- `dnt_ignored` — игнорируется заголовок `DNT` / явный отказ от трекинга.

Если в будущем понадобятся новые сюжеты — добавляется значение в enum схемы **и** соответствующий React-компонент во фронт. Без второй части — fail-fast валидация в загрузчике корпуса.

В MVP `evidence_template` проставляется только для 6 показательных нарушений (по одному на каждый шаблон) — достаточно для демонстрации UX. Остальные 94 нарушения рендерятся через `GenericEvidence` без потери информации.

### 4. SSE-прогресс — вариант B

Engine остаётся как есть: публикует `scanner_started`, `scanner_done`, `violation_evaluated`, `done`, `error`. Шагов прогресса в SSE-стриме **нет**. Фронт группирует `violation_evaluated` по `category` закона на лету (через bundled JSON, генерируемый из [docs/laws/index.yml](../laws/index.yml) на этапе `pnpm prebuild`).

Это означает, что для группировки достаточно `category` на уровне `Law` — добавлять её на уровне `Violation` или вводить отдельный группировщик в engine не требуется.

## Рассмотренные альтернативы

### Вариант A: бэк публикует `step_started` / `step_done`

Engine группирует violations по `category` и эмитит границы шагов в SSE. Контракт SSE становится точнее (фронту не нужно знать категории заранее), но:

- Требует правки `app/engine.py` и `app/events.py` — +1 пункт в DoD итерации 6, расширение тестов.
- Делает engine осведомлённым о UI-понятии «шаг прогресса», тогда как сейчас он работает в терминах «нарушение». Это размывает границу слой и нарушает принцип «engine не знает о UI».
- Дублирует знание о категории: оно есть в YAML, и теперь дополнительно в порядке эмиссии событий.

Отвергнуто. Если в будущем понадобится более точный прогресс — это решение можно ввести отдельной итерацией, без переделки фронта.

### Расширение `violation_evaluated` полями `index` / `total`

Минимальная правка бэка: добавить порядковый номер violation и общее число в payload каждого события. Прогресс-бар становится линейным по числу проверок.

Отвергнуто: концепции «шага» / «категории» это не вводит, а на странице прогресса дизайн именно категориальный (5–6 шагов с иконками, а не один бесшовный прогресс-бар).

### `category` на уровне `Violation`, а не `Law`

В 152-ФЗ есть и privacy-нарушения, и cookies-нарушения; на уровне закона эта неоднородность теряется.

Отвергнуто для MVP: при варианте B все cookie-нарушения 152-ФЗ группируются в шаг `privacy` — это **не критично** для UX (шаги динамические, в MVP их всё равно 5, а не 6). Добавлять category на уровне violation — лишний усложняющий слой; при необходимости вводится позже как опциональное переопределение.

### Хардкод-маппинг в коде фронта

Оставить YAML как есть; держать `LAW_META: Record<law_id, {icon, category, ...}>` в `frontend/src/data/`. Так и сделано в прототипе.

Отвергнуто как долгосрочное решение: дублирует знание (правка категории закона → правка YAML + правка фронта), нарушает «корпус — источник правды», ломается при добавлении нового закона (легко забыть обновить фронт). Допустимо только как временное состояние до этого ADR.

## Последствия

### Положительные

- Корпус остаётся единственным источником правды о праве: добавление нового закона = один YAML-файл + регенерация индекса; UI подхватывает иконку/категорию/описание автоматически на этапе сборки.
- Контракт SSE остаётся стабильным — engine не пришлось переделывать, итерация 6 фокусируется на фронте.
- Закрытые словари (`category` enum, `evidence_template` enum) защищают от опечаток и расхождений между YAML и UI — валидация на уровне Pydantic-моделей.

### Отрицательные / риски

- 15 YAML-файлов нужно обновить вручную (одноразовая работа, маппинг зафиксирован в tasklist'е итерации 6). Митигация: маппинг согласован заранее, правка одного блока в каждом файле.
- Когда понадобится добавить новый `evidence_template` — синхронизация в двух местах (схема + фронт-компонент). Митигация: fail-fast валидация на стороне фронта (`<EvidencePreview template={...}>` рендерит `GenericEvidence` при неизвестном template, не падает).
- Категория `cookies` в MVP-корпусе не используется (нет закона). Карточка на главной всё равно рисуется — это разрыв «корпус ↔ витрина», но карточек ровно 6 по дизайну, а законов в этой нише в РФ пока нет. Митигация: enum резервирует значение, при появлении отдельного cookie-акта он сразу попадает в шаг прогресса.
