# ADR-0001 — Структура и формат справочника российского законодательства

- **Статус:** Accepted
- **Дата:** 2026-05-13
- **Автор:** vadimwork011@gmail.com (совместно с Claude)
- **Связанные документы:** [docs/idea.md](../idea.md), [docs/laws/schema.md](../laws/schema.md)

## Контекст

Проект **Legal_site** — веб-приложение для проверки сайтов на соответствие законодательству РФ (см. [docs/idea.md](../idea.md)). Пользователь вводит URL → приложение анализирует страницы → выдаёт отчёт со списком нарушений, статьями закона, штрафами и рекомендациями.

Чтобы приложение могло детектировать нарушения, ему нужен **источник правды** — структурированная база знаний по законам РФ, требованиям к сайтам, типовым нарушениям и штрафам.

Эта база должна удовлетворять нескольким разнородным требованиям одновременно:

1. **Читаемость людьми** — юристы и разработчики должны видеть и править содержимое без специальных инструментов.
2. **Машинная обработка** — движок проверки должен парсить правила детекции и применять их к страницам сайта.
3. **Удобство обновления** — российское законодательство меняется часто (только в 2024–2025 годах — 420-ФЗ, 168-ФЗ, 121-ФЗ и др.); правка одного закона не должна вызывать конфликты в других.
4. **Аудируемость** — для каждого факта (дата, сумма штрафа, редакция статьи) должен быть указан источник; должна быть видна дата последней верификации.
5. **Расширяемость** — список законов будет расти от MVP-набора к отраслевым.

На момент решения проект пуст: есть только [docs/idea.md](../idea.md), кода и справочников нет.

## Решение

Принят набор взаимосвязанных решений:

### 1. Формат хранения — YAML-фронтматтер + Markdown

Каждый закон / нормативный акт = один файл в `docs/laws/`. YAML-фронтматтер содержит структурированные данные (метаданные акта, перечень нарушений с правилами детекции, штрафы), тело Markdown — человеко-читаемое описание (Кратко / Кому применяется / Ключевые требования / Изменения 2022–2026 / Связанные акты).

Подробная схема — в [docs/laws/schema.md](../laws/schema.md).

### 2. Охват MVP — базовый + e-commerce/онлайн-расчёты

В MVP входят **15 нормативных актов + 2 сводных обзора** (статьи КоАП и регулирование cookie).

**Базовые (любой сайт):**
- 152-ФЗ «О персональных данных» — с учётом 266-ФЗ от 14.07.2022 (уведомление РКН) и 420-ФЗ от 30.11.2024 (оборотные штрафы до 500 млн ₽, в силе с 30.05.2025)
- 242-ФЗ (локализация ПДн)
- ПП РФ № 1119 (защита ПДн)
- 149-ФЗ «Об информации, ИТ и защите информации» — вкл. ст. 10.1 (ОРИ), 10.4, 10.6, 15.1–15.7
- Закон РФ 2300-1 «О защите прав потребителей» — с учётом изменений по 168-ФЗ от 24.06.2025
- 38-ФЗ «О рекламе»
- Маркировка интернет-рекламы (ст. 18.1 38-ФЗ, введена 347-ФЗ от 02.07.2021)
- 436-ФЗ «О защите детей от информации»
- 53-ФЗ «О государственном языке РФ» + 52-ФЗ от 28.02.2023 + 168-ФЗ от 24.06.2025
- Часть IV ГК РФ (авторское право, товарные знаки)

**E-commerce / онлайн-расчёты:**
- ПП РФ № 2463 (Правила продажи)
- 54-ФЗ (ККТ)
- 161-ФЗ (НПС)
- 63-ФЗ (электронная подпись)
- Ст. 437–438 ГК РФ (публичная оферта)

**Сводные обзоры:** статьи КоАП РФ (6.17, 7.12, 13.11, 13.27, 13.34, 14.3, 14.4.1, 14.5, 14.8, 14.15) и регулирование cookie в РФ.

### 3. Поля детекции в каждом нарушении — структурированные правила

Каждое нарушение в `violations:` содержит блок `detection`:
- `page_signals` — признаки на уровне страницы (CSS-селекторы, HTML-паттерны, требуемые/отсутствующие элементы)
- `site_signals` — признаки на уровне сайта (наличие/отсутствие страниц по ключевым словам, заголовкам)

Это позволяет справочнику быть одновременно базой знаний *и* источником правил для движка проверки, без дублирования.

### 4. Источники для агентов — официальные и регуляторы

Приоритет источников при сборе (агент должен сверять каждый факт через ≥ 2 источника):

1. pravo.gov.ru / publication.pravo.gov.ru — официальный портал
2. КонсультантПлюс
3. Гарант
4. Сайты регуляторов: rkn.gov.ru, fas.gov.ru, rospotrebnadzor.ru, nalog.gov.ru

Юридические разборы (Право.ру, Закон.ру) и судебная практика в MVP не используются — слишком высокий риск устаревших или интерпретативных данных.

### 5. Стратегия сбора — батчи по 3–4 агента параллельно

- **Шаг 0.** Эталонный 152-ФЗ собирается первым (он сложнее всех и задаёт стандарт для остальных).
- **Шаг 1.** Остальные 14 актов + 2 сводных обзора собираются батчами по 3–4 параллельных агента. Между батчами — короткий отчёт пользователю.
- **Шаг 2.** Консолидация: генерация `index.yml` из фронтматтеров, заполнение `README.md`.
- **Шаг 3.** Верификация: валидность YAML, обязательные поля, целостность ссылок.

Один агент = один закон = один файл. Это даёт изоляцию ошибок, простой повтор отдельного закона при проблеме и параллельность.

### 6. Структура каталога

```
docs/
  idea.md
  adr/                                 # эта папка — архитектурные решения
    README.md
    0001-laws-reference-structure.md
  laws/
    README.md                          # навигация и инструкция по обновлению
    schema.md                          # формальное описание YAML-схемы
    index.yml                          # машинный индекс (генерируется)
    152-fz-personal-data.md
    149-fz-information.md
    ...
    common/
      koap-overview.md
      cookies-in-russia.md
```

### 7. Стабильные идентификаторы (kebab-case)

ID акта в YAML (`id: 152-fz`) и в файловом имени (`152-fz-personal-data.md`) **не зависят от редакции и даты**. Ссылки `related: [152-fz]` и `[[152-fz]]` остаются валидными при любых правках.

### 8. Маркер свежести `verified_at`

В каждом файле обязательное поле `verified_at` (ISO 8601). Файлы старше 6 месяцев — кандидаты на перепроверку. На следующем этапе появится линтер, который будет это подсвечивать.

### 9. Сознательно отложено в следующую итерацию

- **181-ФЗ + ПП РФ № 102 от 07.02.2026** — доступность сайтов для инвалидов (для гос-сайтов обязательно с 01.03.2026, для коммерции пока рекомендательно)
- **187-ФЗ** — безопасность КИИ
- **236-ФЗ** — «приземление» иностранных платформ
- **374-ФЗ** — «Яровая»
- **422-ФЗ** — самозанятые
- Маркировка товаров «Честный знак»
- Отраслевые: медицина / БАД / лекарства, финансы / кредитование, алкоголь, азартные игры, СМИ, иноагенты

## Рассмотренные альтернативы

### Один большой Markdown-файл

Все законы в одном `reference.md`. **Отвергнуто:** трудно парсить машиной, конфликты в git при параллельной правке, плохо масштабируется.

### Чистый JSON или YAML без Markdown

Все данные в `laws.json` / `laws.yaml`. **Отвергнуто:** теряется человеко-читаемость, юристам неудобно править, нет места для развёрнутых описаний и истории изменений.

### Без полей детекции (только юридическая база)

Справочник — чистая база знаний, логика детекции живёт отдельно в коде движка. **Отвергнуто:** приводит к дублированию (нарушение описано в законе → правило детекции дублирует это описание в коде), при изменении закона нужно править в двух местах, рассинхронизация неизбежна.

### Полный охват с отраслевыми законами с самого начала

Включить в MVP медицину, финансы, алкоголь, азарт, СМИ и т. д. **Отвергнуто:** объём работы кратно вырастает, ценность для типичного пользователя в MVP невысокая (большинство сайтов — общего профиля). Отраслевые добавим итерацией после MVP.

### База данных вместо файлов

Хранить справочник в SQLite / Postgres с админкой. **Отвергнуто на этапе MVP:** нет версионирования через git, нет diff-ов, нельзя ревьюить правки через PR, поднимать инфраструктуру ради 15 актов — перебор. Возможно вернёмся к этому позже, если правки начнут идти потоком.

## Последствия

### Положительные

- Каждый закон правится изолированно — чистый git diff, простой code review.
- Один формат для людей и машины — нет дублирования между «документацией для юристов» и «правилами для движка».
- Стабильные ID позволяют делать сильные кросс-ссылки между актами.
- Поле `verified_at` даёт инструмент управления свежестью базы.
- ADR-папка даёт место для фиксации будущих архитектурных решений.

### Отрицательные / риски

- YAML с глубокой вложенностью (`violations[].detection.page_signals[].html_patterns`) — нюанс при ручной правке, легко сломать отступы. Митигация: валидация в линтере.
- Файлы могут вырасти большими (эталонный 152-ФЗ получился 754 строки) — нагрузка на review. Митигация: один файл = один закон, никто не читает всё подряд.
- Зависимость от живых источников (pravo.gov.ru, КонсультантПлюс) — если меняется структура их сайтов, агент-сборщик может ломаться. Митигация: фиксируем найденные URL в `official_sources`, при следующей верификации перепроверяем доступность.
- При наличии 420-ФЗ в МАЕ 2026 года точные номера частей ст. 13.11 КоАП РФ местами реконструированы из вторичных источников — нужна сверка с pravo.gov.ru при первой возможности (TODO зафиксированы в файле 152-ФЗ).

## Ответственное обращение с неполной верификацией

Часть данных в справочнике объективно невозможно подтвердить через открытые
источники (платный КонсультантПлюс/Гарант, не отвечающий WebFetch с
pravo.gov.ru, неопубликованные тексты судебных решений). Такие границы
зафиксированы в одном машиночитаемом документе —
[docs/laws/common/data-coverage-limitations.md](../laws/common/data-coverage-limitations.md).

Принципы:

- **Не скрывать ограничения.** Каждый `caveat` имеет `affects_violation_ids`
  и `affects_subjects` — движок проверки сайтов обязан вывести оговорку
  в отчёт по сайту в карточке соответствующего нарушения.
- **Указывать, куда идти за подтверждением.** Каждый `caveat` содержит
  `sources_to_confirm` — pravo.gov.ru, платные подписки, конкретные
  судебные сервисы.
- **`severity` оговорки** — `medium` означает заметную пометку рядом
  с суммой штрафа, `low` — сворачиваемую сноску.
- **Жизненный цикл:** когда оговорка снимается (данные подтверждены), запись
  из `caveats[]` удаляется, в `review_log` соответствующего закона
  фиксируется источник верификации.

Это решение распространяет принцип «один источник правды» с фактических
норм на **метаданные о точности норм**: и юристу, и пользователю сайта,
и движку видна одна и та же картина того, что верифицировано до конца,
а что — оценочно.
