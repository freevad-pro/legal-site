# Architecture Decision Records (ADR)

В этой папке хранятся записи о значимых архитектурных и проектных решениях. Каждый ADR описывает контекст, само решение, альтернативы, последствия и при необходимости — историю изменения статуса.

## Зачем

- **Память проекта.** Через 6 месяцев никто не помнит, *почему* выбрали тот или иной формат / стек / подход. ADR отвечает «почему».
- **Онбординг.** Новый участник читает ADR подряд и быстро входит в курс.
- **Откат без потери контекста.** Если решение оказалось плохим, можно посмотреть, какие альтернативы рассматривались и что заставило выбрать именно это.

## Формат

Один ADR = один файл `NNNN-краткое-название.md`, где `NNNN` — порядковый номер (0001, 0002 …). Внутри:

- **Статус** — `Proposed` / `Accepted` / `Deprecated` / `Superseded by ADR-XXXX`.
- **Дата** — принятия в ISO 8601.
- **Контекст** — что мы делали и почему вообще понадобилось решение.
- **Решение** — что именно решили (короткая формулировка).
- **Рассмотренные альтернативы** — что отвергли и почему.
- **Последствия** — что меняется в проекте после принятия, плюсы и минусы.
- **Связанные ADR / документы**.

## Индекс

- [0001-laws-reference-structure.md](0001-laws-reference-structure.md) — Структура и формат справочника российского законодательства для проверки сайтов.
- [0002-corpus-ui-fields.md](0002-corpus-ui-fields.md) — Расширение схемы корпуса полями для UI (`category`, `icon`, `short_description`, `evidence_template`).
- [0003-context-gating-and-inverse-keywords.md](0003-context-gating-and-inverse-keywords.md) — Контекстный гейтинг нарушений через `applicability` и инверсная семантика `prohibited_keywords`; скрытие заглушек через `inconclusive_reason`.

## Когда добавлять новый ADR

- Выбор стека / библиотеки / формата данных.
- Изменение архитектуры или схемы данных.
- Решение, которое неочевидно и которое потом будет дорого «откатить».
- Изменение охвата проекта (что в MVP, что нет).

Если решение тривиально и сразу очевидно — ADR не нужен.

## Когда менять существующий ADR

- ADR **не редактируется** ретроспективно (это история решения).
- Если решение пересмотрели — создаём новый ADR со статусом `Accepted` и в старом ставим `Superseded by ADR-XXXX`.
- Если решение признано неудачным и отменено без замены — ставим в старом `Deprecated` и кратко описываем почему.
